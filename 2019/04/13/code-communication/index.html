代码写好很重要，与人交流沟通也很重要

自己工作了几年，写了几年代码，发现一个很有趣的现象，许多工程师经常是能够在同是一线工程师的很容易交流沟通，发现一但跨出工程师的工种之后，都非常少的交流，甚至有人都会存在交流恐惧的现象。甚至有时候与人交流的时候，很容易有心理恶意猜测的现象，比如：

  • 这么傻的问题，怎么还会来问我？
  • 我只是写代码的，你问得许多问题问我Leader或者去问PM去。
  • 这个问题我和他解释，我说了，他也不懂的，还是免得废话了。

一方面，有些工程师会觉得我只是来写代码的，与人打交道，交流不是我的工作。另外一方面，有些工程师又是代码写起来特别好，但与人交流有时候真的比较困难，想交流，但经常不知道怎么表达。

工程师真的只需要写好代码吗？

关于，这个问题我是这样看的。工程师必须需要写好代码，但绝对不是只需要写好代码（感觉就是一句废话）。

试想下，你平时自己工作中，代码的需求来源都是一个个与人交流过来的结果，简单来说代码的需求都是来自于人。再想下，你的代码完成之后，使用产品的人，也是一个个不同的人。再比如，每次你看别人写得代码，看源码库，背后的作者也都是一个个人。甚至有时候，发现几乎所有的东西都是来自于人打交道交流的结果。

当然上面说得与人交流沟通很重要，许多工程师也都意识到，只不过经常觉得自己做起来好难。经常是，我知道这个很重要，但我不知道怎么做好这个，所以最后我放弃了。

试着一对一交流

我们不是为了做演说家，给几百人做演讲。事实上，我们甚至一辈子也不会有几百人做演讲的机会。所以先可以试着一对一的找一个人来沟通交流。

写代码的时候，大家都会从Hello World！开始学习，先从最小的一个主函数开始学习，再慢慢的深入。所以与人交流，先从最简单的开始，只找一个人来聊。许多人接着会遇到聊什么的问题，直接找个同事聊聊工作，聊聊你最近做得什么工作。写前端的，找个设计师聊聊最近做得功能。写后端的、找个PM，聊聊做得业务逻辑，问问未来的发展方向。做算法应用的，找个数据PM，聊聊最新的数据落地情况……总之，先勇敢的试着就找一个人聊，把范围变小，坚持的去沟通，相信很快会有收获。

看看非技术书

许多研发同学，尤其是刚毕业的同学，经常会看各种技术书, 什么编程思想，什么实战等等，几百页都能随便啃下来。

推荐有空，业余时间看看非技术书籍。一般这些书籍读起来都特别快，比起那些晦涩难懂的技术数据，这些书籍可能半天就能一本开完。这些书籍多去看看，能够让自己多开开眼界思路。这个世界，程序员毕竟少数，更多数的人看得是心灵鸡汤、听得是郭德纲，所以试着去看看非技术书籍，了解了解别的世界。

勇敢承认错误

平时在与人交流过程中，许多工程师在被别人指出错误的时候，经常会先马上进行反驳。尤其是许多时候，用户不是按照设定的流程在使用，然后指出bug，极度容易引起工程师的反感，甚至有时候还会发生争吵。这时候勇于先承认错误，避免进一步争吵，先把问题解决了。说一句“我错了”的力量，往往有意向不到的效果。

许多时候，当你先承认错误之后，事后甚至会发现获得了很多的尊敬。在公司里面，尤其是积极向上的公司，一般讲究的文化都是解决问题的文化。错误是谁犯得，可能没有那么的重要，是谁解决的才重要。所以勇于承认错误，然后积极的把事情解决吧。

当然与人打交道的学问还有很多，比如面对上司如何沟通、面对下属有人如何沟通、和完全不同行业的怎么沟通？这些都是可以持续学习的地方。
